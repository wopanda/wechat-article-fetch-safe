# 阶段总结 v1：wechat-article-fetch-safe

## 当前定位

这份仓库当前应被视为一个 **微信公众号文章抓取底座 v1**，而不是一个追求 100% 成功率的终极抓取器。

它的职责是：
1. 接收公众号文章 URL
2. 优先使用 HTTP 主链提取正文
3. 在必要时切换到浏览器 fallback
4. 返回结构化结果与失败原因
5. 为上层 Agent / Skill / MCP 提供稳定输入

## 当前已实现能力

### 抓取链路
- 微信移动 UA HTTP 提取
- 桌面 UA 对照提取
- Playwright 浏览器 fallback
- 浏览器滚动与简单稳定性判断
- OCR 最后兜底（仅在浏览器页面可见但 DOM 提取明显不足时尝试）

### 提取内容
- 标题
- 作者（多数场景）
- 封面图
- 正文 Markdown
- 图片链接列表
- 部分发布时间

### 工程能力
- 结构化失败分类
- `page_signals` 页面异常识别
- `quality_metrics` 质量指标输出
- debug artifacts（HTML / meta / screenshot）
- 最小回归框架与摘要统计

## 当前明确不做的事

- 不承诺任意文章 100% 成功
- 不自动突破 verify / captcha / anti-bot
- 不自动写入飞书、知识库、数据库等外部系统
- 不把所有个别文章特例都堆进主链

## 当前阶段结论

### 已成立的判断
1. 这条技术路线是成立的：**HTTP 优先 + Browser fallback + OCR 最后兜底 + 失败分类**
2. 当前仓库已经具备继续作为上层能力底座使用的条件
3. 不应该继续在抓取细节上长期深挖，否则会进入边际收益快速下降区

### 未完成但不阻塞进入下一步的
- 发布时间提取仍不稳定
- verify / anti-bot 样本仍偏少
- 个别文章尾部清洗还可以更细
- OCR 阈值与触发条件仍可继续校准

这些问题目前都 **不构成阻塞项**。

## 为什么现在切下一步

因为当前最缺的已经不是脚本内部逻辑，而是：
- 上层接入与真实场景使用
- 更合理的验证分层
- 更稳定的输出协议与交付边界

继续大量优化选择器、噪音规则、发布时间细节，会显著增加过拟合风险，但不会成比例提升整体价值。

## 下一步建议

### 建议立即进入
- 能力定型
- 输出协议冻结
- 上层接入（Skill / Agent / MCP / 内容流程）

### 建议暂缓
- OCR 主链接入
- 大规模规则补丁
- 针对单篇文章的特例优化

## 推荐验收口径（v1）

当前版本只需要满足以下标准：
1. 能对一部分可公开公众号文章稳定返回正文 Markdown
2. 失败时能返回明确状态，而不是无结构报错
3. 必要时可提供 HTML / screenshot 调试证据
4. 能被上层系统稳定调用

达到这四条，就足够进入下一步。
